## What Is LogHound?

LogHound is an iOS 18+ app for creating fully custom tracking logs. Unlike apps with fixed templates (symptom trackers, habit apps, food diaries), LogHound lets the user define their own schema: they choose the fields, the field types, and the field names. Once a schema exists, they record entries against it and view their data three ways: as scrollable cards, as a sortable table, or as an aggregated summary.

Everything is stored on-device using Apple's SwiftData framework. There is no cloud sync, no account creation, no internet connection required at any point, and no data ever leaves the device.

LogHound includes a 30-day free trial. After the trial, a one-time purchase unlocks the app permanently — there is no subscription.

## Core Concepts (Read First)

Before answering specific questions, understand these concepts — they explain the "why" behind many user questions:

**Log**: A named container with a defined schema (set of fields) and a collection of entries. Analogous to a spreadsheet file.

**Schema / Fields**: The structure of a log — which fields it has and what type they are. Defined once in the schema editor, but editable at any time. Changing the schema affects future entries; existing data is retained (with caveats for field deletion).

**Entry**: A single record in a log. Has a timestamp and a value for each field in the schema. Analogous to a row in a spreadsheet.

**Field Type**: Determines what kind of data a field stores and how it's displayed and edited. Once set, the type cannot be changed — the field must be deleted and recreated.

**Views**: Three ways to look at a log's entries — Cards (individual records), Table (spreadsheet-style grid), Summary (aggregated statistics over time).

**Aggregatable fields**: Number, Rating, Toggle, Duration, and Distance/Pace — these are the only types that can be summarized in the Summary view. All other field types are excluded from aggregation.

## First Launch Experience

On first launch the app shows a **welcome screen** ("Welcome to LogHound") with two choices: **Get Started** (begins the guided walkthrough) and **Skip for now** (goes straight to the empty log list).

The walkthrough:
- Explains the difference between a **Log** (the container with a schema) and an **Entry** (a single record)
- Runs a short **live demo** of building a schema and recording a first entry
- Can be paged with **Back / Next** and left at any point via a **Skip** button
- At the end, offers to either create a ready-made sample log called **"My Morning Routine"** or start from scratch with the user's own log

The walkthrough can be **replayed at any time** from **Settings → Help → Getting Started**.

If the user chooses the sample, the single **"My Morning Routine"** log is created, pre-seeded with **one example entry** so they can immediately explore cards, table, summary, and the schema editor. It can be explored, edited, or deleted just like any user-created log. After it's created, a one-time hint banner ("Tap '[log name]' to open it") appears atop the log list.

## Creating and Managing Logs

### Creating a New Log

Tap **+** (top-right of log list) → in the "New Log" sheet enter a name → tap **Create**.

The schema editor opens immediately so the user can add fields. The log exists even before any fields are added — a log with no fields shows a distinct empty state ("This log only records timestamps right now…") with an **Add Fields** action, since without fields it just records timestamps.

**Name rules**: Must be unique among active logs. Maximum length is not explicitly limited but very long names will be truncated in the UI.

### Renaming a Log

"..." menu → **Rename** → edit the name → confirm.

Will reject the new name if it duplicates another active log's name. The name does not need to be unique across stashed or trashed logs.

### Duplicating a Log

"..." menu → **Duplicate** → choose a new name → set options → tap **Duplicate**.

Options:
- **New name**: Pre-filled with "[original] Copy". Must be unique.
- **Include entries** toggle (default: **OFF**): If on, all entries from the original are copied into the new log. If off (the default), only the schema is copied — the new log starts empty.

After duplicating, the new log opens in the schema editor so the user can make adjustments.

**Use case**: Great for creating a log with the same structure but starting fresh (e.g., a new month's log with the same fields as last month).

### Copy Fields to New Log

Available from the log detail "..." menu (not the log list). Creates a brand new log with identical fields but no entries. Opens the schema editor for the new log. Useful when the user wants a similar structure but wants to customize it first.

**Difference from Duplicate**: Duplicate always uses the existing schema as-is and optionally copies entries. Copy Fields always starts with no entries and opens the schema editor for customization.

### Stashing a Log

"..." menu → **Stash**.

The log disappears from the active log list and moves to the Stash. All data is fully preserved. Stashed logs do not appear in search results.

To access stashed logs: tap the **Stash button** (bottom-left toolbar) on the log list.

The Stash view has its own **search bar** ("Search stash") and each row shows the entry count plus a "Stashed [date]" line.

To restore: open the Stash → tap the log → tap **Restore**. If the name conflicts with an existing active log, a **Rename Log** sheet (a name-conflict prompt, pre-filled with the current name) appears asking the user to choose a new name before confirming with **Restore**.

**Stash vs Delete**: Stash is for logs you're done with but want to keep. Delete (trash) is for logs you no longer need, with a 30-day safety window before permanent deletion.

### Deleting a Log (Sending to Trash)

"..." menu → **Delete**.

The log moves to Trash. It remains there for **30 days** before being automatically and permanently deleted.

To access trashed logs: tap the **Trash button** (bottom-right toolbar).

The Trash view shows each log with a countdown ("N days left", which turns **red at 7 days or fewer**) and a "Deleted [date]" line. From here the user can:
- **Restore**: moves back to active logs (with name-conflict handling)
- **Delete Now** (per-log): immediate permanent deletion, after a "Delete Permanently" confirmation
- **Empty Trash** (button at top): permanently deletes all trashed logs at once

### Delete All Entries

From the log detail "..." menu → **Delete All Entries**. Permanently deletes every entry in the log but leaves the log itself and its schema intact. Requires confirmation showing the exact entry count. This cannot be undone.

**Use case**: Starting fresh with a new period of tracking while keeping the same schema.

- "..." menu in the log detail view → **Edit Fields**
- Automatically opened after creating or duplicating a log

### What's on Screen

- **Title**: "Logging Setup" with a subtitle "What You're Tracking"
- List of existing fields, each showing:
  - A type icon (left)
  - Field name

- **"+ Add Field"** button at the bottom

Drag fields using the handle on the right side. The order determines:
- The order fields appear in the entry form
- The order of columns in Table view (timestamp is always first)

### Deleting a Field

Tap the red minus button on the left of a field row, or swipe left on the field.

A confirmation dialog shows:
- How many existing entries have data in this field
- A warning that this data will be permanently lost

Tap anywhere on a field row (not the minus or drag handle) to open the field configuration editor.

Once a field is created with a specific type, the type cannot be changed. The user must delete the field and add a new one with the desired type. They will lose any data stored in the old field.

### Schema Preview

The eye icon (top-right) opens a **FormPreviewView** that shows exactly how the entry form will look with the current fields. This is useful for checking layout and order before committing.

### "Done" Button

Tap Done (top-right) to close the schema editor and return to the log detail view.

## Field Types — Complete Reference

### 1. Text

**What it stores**: Any string of characters. Supports emoji, line breaks, and special characters.

**Entry form**: Standard iOS text input. Keyboard appears when focused. A "dismiss keyboard" (×) button appears in the keyboard toolbar.

**Empty vs filled**: An empty string (nothing typed) and whitespace-only entries are treated as empty/unfilled.

**Configuration options**:
- **Default value** (optional) — pre-fills new entries with this text
- **Required** — see Field Configuration Reference
(There is no maximum-length setting.)

**Display in views**:
- Cards: shown as text, wraps to multiple lines if long
- Table: shown truncated to fit column width; tap row to see full value
- Summary: Text fields do not appear in Summary view

**When to use**: Notes, descriptions, labels, freeform observations, anything that doesn't fit a structured type.

---

### 2. Number

**What it stores**: A numeric value. Can be integer or decimal depending on configuration.

**Entry form**: Numeric keyboard. The user types a number. Min/max validation is shown inline — if the value is out of range, a warning appears and the Save button may be disabled.

**Zero is valid**: A value of 0 is considered filled, not empty. A required Number field is satisfied by entering 0.

**Negative values**: Allowed unless a minimum value prevents it.

**Configuration options**:
- **Allow decimals** (toggle) — if OFF, only whole numbers are accepted; if ON, decimal values like 5.2 or 3.14 are accepted. Affects keyboard type.
- **Decimal places for summaries** (stepper, 0–4, default 1) — a **per-field** setting controlling how many decimals this field's aggregated values show in Summary view. This is separate from the global Table-view precision setting.
- **Minimum value** (optional) — input is rejected if below this
- **Maximum value** (optional) — input is rejected if above this
- **Default value** (optional) — pre-fills new entries with this number

**Display in views**:
- Cards: shown as-is
- Table: shown with the number of decimal places set in Settings (Numeric Precision, 0–10). This is purely display — stored precision is unaffected.
- Summary: appears as a column; participates in Sum, Average, Count. Decimals shown follow this field's own "Decimal places for summaries" setting (0–4), not the global Table precision.

**When to use**: Distances, durations, weights, measurements, counts, calories, anything quantitative.

---

### 3. Toggle

**What it stores**: A boolean yes/no value.

**Entry form**: A switch labeled "Yes" / "No". Defaults to **No (false)** in new entries — but this can be flipped with the **"Default to On"** configuration option so new entries start at Yes.

**Always filled**: Unlike other field types, a Toggle always has a value (either yes or no). It cannot be empty. Because of this, **Toggle fields cannot be set as Required** — the setting has no effect since the field is inherently always filled.

**Configuration options**:
- **Default to On** (toggle) — if enabled, new entries default to Yes instead of No

**Display in views**:
- Cards: shown as "Yes" or "No"
- Table: shown as "Yes" or "No"
- Summary: Toggle contributes 1 (Yes) or 0 (No) to Sum and Average calculations. With Sum + Day, you get a daily count of how many Yes entries there were. With Average + Week, you get a weekly compliance rate (e.g., 0.71 = completed the habit 5 out of 7 days).

**When to use**: Habit tracking (did I do X today?), yes/no attributes (took medication, completed warmup, paid bill).

---

### 4. Single Select

**What it stores**: One selected option from a predefined list.

**Entry form**: A scrollable list of options. Selecting one deselects any previous selection. Can be left empty (no option selected).

**Option management**: Options are defined in the field configuration. Each option can be:
- **Active**: appears in the picker for new entries
- **Removed (deprecated)**: hidden from the picker when creating or editing entries, but preserved in existing entries that used it. Deprecated options still display correctly in card/table views for old data.

To remove/deprecate an option: edit the field in the schema editor → **swipe left** on the option (or **long-press** it) → **Remove**. Removing does **not** delete the option or destroy old data — it moves the option into a **"Removed Choices"** section at the bottom of the option list, each with a **Restore** button to bring it back. There is no hard-delete of an option that would erase historical values. Adding or renaming an option is blocked if the name case-insensitively duplicates an existing active option ("This option already exists").

**Configuration options**:
- **Options list**: add, rename, reorder, and remove/restore options (as above)
- **Default selection** (optional): one option (or None) is pre-selected in new entries

**Display in views**:
- Cards: shown as the option text
- Table: shown as the option text
- Summary: Single Select does not appear in Summary view

**When to use**: Category, route, location, status, type — anything with a fixed set of mutually exclusive choices.

**Single Select vs Multi Select**: Use Single Select when exactly one option applies. Use Multi Select when multiple options can apply simultaneously (e.g., symptoms present, focus areas in a practice session).

---

### 5. Multi Select

**What it stores**: Zero or more selected options from a predefined list.

**Entry form**: A **chip-style** picker — options appear as tappable chips, and tapping a chip toggles it on or off. Multiple chips can be selected at once. A count of selected options is shown. Can be left empty (no chips selected).

**Option management**: Same as Single Select — options can be added, renamed, reordered, and removed/restored (removed options move to a "Removed Choices" section, preserved in existing entries). A **default set** of pre-checked options can also be configured for new entries.

**Display in views**:
- Cards: selected options shown as **chips**
- Table: selected options shown as a comma-separated list (e.g., "Neck, Shoulders"), truncated in narrow columns
- Summary: Multi Select does not appear in Summary view

**When to use**: Tags, multiple applicable categories, symptoms (multiple can be present at once), focus areas, ingredients, skills practiced.

---

### 6. Date/Time

**What it stores**: A calendar date together with a time of day.

**Entry form**: Shows a combined date and time picker. Always displays:
- The selected date/time value
- A **relative time indicator** alongside the value (e.g., "2 hours ago", "in 3 days", "yesterday", "in 4 wk 5 days")
- A **Clear** button — removes the value entirely (field becomes empty)
- A **Now** button — instantly sets the field to the current date and time

**Configuration options**:
- **Default time offset** (toggle "Count from entry timestamp"): if set, when creating a new entry this field auto-populates to (entry timestamp) + (offset). The offset is built from **months, days, hours, and minutes**, and can point **before or after** the entry timestamp (a +/− direction). E.g. "+1 hour" defaults the field to 1 hour after the entry; "−3mo 15d" to about three and a half months before. Useful for "next scheduled time" or "previous checkup" patterns. (This is the only configuration option — there is no date-only / time-only mode; the field always captures both date and time.)

**Display in views**:
- Cards: shown as formatted date/time or relative time
- Table: shown as formatted date/time
- Summary: Date/Time fields do not appear in Summary view (cannot be meaningfully aggregated)

**When to use**: Appointments, due dates, scheduled times, "next recital", event logging with precise times, dose times, medication scheduling.

---

### 7. Duration

**What it stores**: A span of time in days, hours, minutes, and seconds, with an optional running timer you can start and stop. Good for workouts, activities, or anything you time.

**Entry form**: An edit sheet with four numeric fields — **Days / Hours / Minutes / Seconds** — plus an optional built-in **timer** that can be started and stopped to capture elapsed time directly.

**Can be empty**: A duration of zero (or no value entered) is treated as empty/unfilled.

**Configuration options**:
- **Table & Summary unit** (Days / Hours / Minutes / Seconds, default Minutes) — the unit used to display this duration in Table and Summary views and to format it in CSV export.

**Display in views**:
- Cards: shown in a compact token form that omits leading zero components — e.g. `"49s"`, `"1m 49s"`, `"2h 30m 16s"`, `"1d 10h 30m 16s"` (not a colon-separated "1:24:30" form)
- Table: shown as a number in the configured Table & Summary unit (e.g., minutes)
- Summary: participates in Sum, Average, Count — aggregates the underlying time value (summed in seconds internally) and displays it in the configured unit

**When to use**: Workouts, activities, tasks, or anything where you want to record how long something took.

---

### 8. Rating

**What it stores**: A numeric rating from 1 to the configured maximum (choices: 3, 4, 5, 6, 7, or 10; default 5).

**Entry form**: Tappable emoji icons arranged horizontally. Tapping a position sets the rating. The selected emoji is repeated to visually represent the value (e.g., 💪💪💪 for a rating of 3).

**Can be empty**: No emoji tapped = empty/unfilled.

**Configuration options**:
- **Maximum rating**: 3, 4, 5, 6, 7, or 10 (default 5)
- **Emoji**: the rating symbol is **customizable** — any single emoji (not necessarily a star). The picker offers positive/negative preset rows plus a custom-emoji option. Examples: ⭐, 💪, 🎵, 😊, 🔥. Default: ⭐
- **Default value** (optional) — a pre-set rating applied to new entries

**Display in views**:
- Cards: repeated emoji (e.g., 🎵🎵🎵🎵 for 4)
- Table: numeric value (e.g., 4)
- Summary: participates in Sum, Average, Count — appears as a numeric column

**When to use**: Quality, effort, mood, satisfaction, perceived difficulty — anything where a nuanced rating is more expressive than a raw number.

---

### 9. Location

**What it stores**: GPS coordinates — latitude and longitude, and optionally altitude and accuracy values.

**Entry form**: A **"Capture Current Location"** button. When tapped:
- The app requests iOS location permission if not already granted
- On success: displays the coordinates to 6 decimal places
- On failure or denied permission: shows an error

**Configuration options**:
- **Include altitude**: stores the device's altitude in meters
- **Include horizontal accuracy**: stores the accuracy radius of the lat/lon in meters
- **Include vertical accuracy**: stores the accuracy of the altitude reading

**Display in views**:
- Cards: shown as "lat, lon" to 6 decimals, with altitude if configured; when accuracy capture is enabled, a second "Accuracy:" line shows `±Xm horiz` / `±Xm vert`
- Table: shown as coordinates
- Summary: Location fields do not appear in Summary view

**Cannot manually type coordinates**: There is no text input — location can only be captured by tapping the button (which reads live GPS).

**When to use**: Recording where something happened (a sighting, a run starting point, a maintenance site), location-tagged observations.

---

### 10. Distance/Pace

**What it stores**: Uses GPS to track distance traveled plus pace and speed. Records as three parts — distance, elapsed time, and pace/speed — ideal for runs, walks, rides, and hikes.

**Entry form**: A GPS-based tracking control that measures distance as the device moves. When recording finishes, the field holds three related values: total distance (with unit), elapsed time, and the resulting pace/speed (with unit). Tracking continues while the app is in the background (a status indicator shows), so a run can be tracked with the screen off — this uses GPS continuously and consumes battery.

**Configuration options**:
- **Distance unit**: Miles (default), Kilometers, or Meters
- **Time display unit**: Hours, Minutes (default), or Seconds
- **Display as** — whether the derived value is shown as **pace** (time per distance) or **speed** (distance per time)

**Display in views**:
- Cards: shown as distance plus pace/speed, with elapsed time
- Table: shown as the recorded distance and pace/speed values
- Summary: participates in Sum, Average, Count — aggregates the underlying values (see GPS accuracy note below)

**How distance is measured (and its limits)**: Distance is computed purely from GPS — the app sums the straight-line (great-circle) distance between consecutive GPS points as you move. Because of this:
- It measures **horizontal (map) distance only** and does not account for **elevation change**, so on **hilly terrain** the true path you traveled up and down slopes isn't fully captured.
- **GPS signal error** — especially in **urban areas** where signals bounce off buildings ("multipath"), and from ordinary GPS jitter — adds spurious little zig-zags between points that can **overestimate** the distance.
- In poor signal, points that fall below an accuracy threshold are dropped, which can occasionally **undercount** distance instead.
- Elapsed time is wall-clock and keeps running even while you're stationary (there is no auto-pause), which affects the resulting pace/speed if you stop.

Treat GPS distance, pace, and speed as **good approximations**, not survey-grade measurements — expect some error, generally largest in hilly and built-up areas. For an exact figure, enter it manually with a plain **Number** field instead.

**When to use**: Runs, walks, rides, hikes — any activity where an approximate GPS distance, time, and pace/speed is useful.

## Field Configuration Reference

Every field type shares these configuration options:

### Name
Any text, including emoji. An **emoji picker** is available inline when editing the name. The emoji appears as a leading icon in the entry form, cards, and table columns.

A field name is **required** (an empty or whitespace-only name is rejected with "Field name is required") and must be **unique within a log** (compared case-insensitively, after trimming). A set of names is **reserved** and cannot be used because they collide with the built-in entry timestamp or location internals: `time`, `timestamp`, `createdat`, `created_at`, `latitude`, `longitude`, `lat`, `lng`, `lon`, `coordinates`, and `location` (all case-insensitive). If a user asks why they can't name a field "Location" or "Time," this is why — they should pick a slightly different name (e.g., "Where" or "Start Time").

### Required
When enabled, the **Save button** in the entry form is disabled until this field has a value. The field label shows a red asterisk (*).

Notes:
- Toggle fields cannot be Required (they always have a value)
- For Number fields: 0 counts as a valid value and satisfies Required
- For Date/Time fields: a date must be set (Clear removes the value and makes the field empty again)
- For Rating: at least one emoji must be tapped

### Copy-on-duplicate behavior
When an **entry** is duplicated, all of its field values are copied into the new entry (only the timestamp is reset to the current time). There is **no per-field toggle** in the app to control this — every field carries its value into a duplicate by default. (If a user wants a different value in the copy, they simply edit that field in the new entry before saving.)

### Default Value
A value pre-filled when creating a **new** entry (not when duplicating — duplicates copy from the source entry instead). Default behavior varies by field type. For Date/Time, the "default time offset" serves as the default value mechanism.

## Recording Entries

> 📸 `assets/screenshots/06-entry-form.jpg`

### Creating a New Entry

Tap the blue **+** floating action button (bottom-right corner of the log detail view). Opens the entry form in "New Entry" mode.

**Timestamp**: At the top, automatically set to the current date and time. Tap **Edit** to open a date/time picker. Changing the timestamp is useful for logging something that happened earlier (e.g., "I should have logged this run yesterday").

**Fields**: One section per field, in schema order. Fill in as many as needed.

**Save**: Tap **Save** (top-right). If required fields are empty, Save is greyed out with an indicator on the unfilled required field.

**Cancel**: Tap **Cancel** (top-left) to dismiss without saving. Values entered so far are saved as a **draft** (see below).

### Drafts

If the user starts filling an entry form and dismisses it without saving, the in-progress values are automatically saved as a draft. The next time a new entry form is opened for the same log, the draft is restored.

- Drafts survive app backgrounding and device restarts
- A draft is cleared when the entry is saved or when the user explicitly discards it
- Only one draft per log exists at a time
- Drafts are per-log — the draft for one log is separate from the draft for another

If a user reports seeing a form pre-filled with old data, it's because a draft was saved from a previous session.

### Editing an Entry

**From card view**: Tap anywhere on the entry card to open the **Edit** form directly. (The only button shown on the card itself is the duplicate icon; **Edit** and **Delete** also live in the card's **long-press context menu**.)

**From table view**: Tap a row to open a detail modal (with Edit and Duplicate), or **long-press / right-click** a row and choose **Edit**.

The edit form is identical to the creation form, plus a red **Delete Entry** button at the very bottom.

### Duplicating an Entry

Tap the duplicate/copy icon on a card (the overlapping pages icon), or from the entry detail view tap **Duplicate**. From table view: right-click → Duplicate.

Opens a new entry form with **every field pre-filled** with the source entry's values. The user can change any of them before saving.

The **timestamp** is set to the current time (not copied from the source entry). The user can edit the timestamp before saving.

### Deleting an Entry

- Card view: **long-press** the card → **Delete** from the context menu
- Edit mode: scroll to the bottom of the form → tap the red **Delete Entry** button
- Table view: right-click/long-press a row → Delete

Deletion is **immediate and permanent** from the user's perspective — there is no undo and no trash for individual entries (only for entire logs).

**Timestamp editor**: When editing an entry's timestamp (or any Date/Time field), the picker includes quick-adjust buttons beyond just **Now** and **Clear** — e.g. a start-of-day ("12am") button and ± step buttons (±15m, ±30m, ±1hr, ±1 day/week/month/year) with a +/− direction toggle, for fast nudging without scrolling the wheels.

### Keyboard Handling

A **dismiss keyboard** button (×) appears in the toolbar above the keyboard when any text input is focused. Tapping it hides the keyboard without moving focus or losing the current input.

### Summary View

> 📸 `assets/screenshots/12-summary-sum-week.jpg`
> 📸 `assets/screenshots/13-summary-avg-week.jpg`

Aggregates the log's numeric data over time intervals.

**Which fields appear**: Only **Number**, **Rating**, **Toggle**, **Duration**, and **Distance/Pace** fields. Text, Single Select, Multi Select, Date/Time, and Location are excluded. If the log has no aggregatable fields, a message says so and the view is empty.

**Two pickers at top**:

**Interval picker** (Day / Week / Month / Year):
- Groups entries into time buckets
- The Date column shows the start of each bucket: "2026-03-15" means the week of March 15
- "Week" respects the **Week starts on** setting (Sunday or Monday) from Settings
- Buckets with no entries are not shown — the table only shows intervals that have at least one entry

**Statistic picker** (Sum / Average / Count):
- **Sum**: adds all values in each bucket. Example: total miles run in a week = 15.2
- **Average**: divides the sum by the number of entries **that actually have a value for that field** (blank/unfilled entries are excluded from the denominator, not counted as zero). Example: average miles per run in a week = 5.1 (if 3 runs).
- **Count**: counts entries regardless of field values. Shows how many times the user logged something in each interval. For Toggle fields, Count tells you how many entries existed; Sum tells you how many were "Yes."

**Notes on specific types**: A stored **Rating of 0** is treated as unfilled and excluded from Sum/Average/Count. **Duration** aggregates in seconds internally and **Distance/Pace** in meters, each converted to the configured display unit. A Distance/Pace field appears in Summary as **three sub-columns** (distance, time, pace/speed); its Sum/Average pace-or-speed is derived from the aggregated totals (total distance ÷ total time), which is the correct combined pace — not a plain average of each entry's pace.

**Toggle fields in summary**:
- **Sum**: number of "Yes" entries in the interval (e.g., 5 out of 7 days I did the habit)
- **Average**: proportion of "Yes" entries (e.g., 0.71 = 71% compliance)
- **Count**: total number of entries in the interval (regardless of Yes/No)

**Filtered count**: When a filter is enabled, the shared Fields & Filters bar shows "**showing N of M**" (N passing the filter out of M total). Summary's own empty states are "No Numeric Fields" (no aggregatable fields) or "No Entries".

**Column sorting**: Tap any column header to sort by that value.

**Fields button**: Toggle which aggregatable fields appear as columns.

**Week start day**: Configured in Settings → Week starts on (Sunday or Monday).

## Search and Filters

### Global Search

The search bar at the top of the log list (placeholder: "Search entries") searches:
- All active log names
- All entry field values across all active logs
- Entry timestamps (formatted text)

Uses **fuzzy matching** — partial matches and slight misspellings are found.

Results appear as a list. Each result shows:
- The log the entry belongs to
- The entry timestamp
- The matching field content with highlighting

Tapping a result navigates to that log in card view and highlights the matching entry.

Search does **not** search stashed or trashed logs.

Clearing the search field (tap × in search bar) returns to the full log list.

### Per-Log Filters

Each log has its own filter configuration that is **independent from other logs** and **persists between sessions**. A filter set on Tuesday is still active on Thursday.

#### Accessing Filters

Expand the **Fields & Filters** bar (in card, table, or summary view) and tap the **Filters** button (labeled "No Filters" / "Filters (N)" / "Filters (off)"). All three views share the same filter sheet for that log.

The **enable/disable** of the whole filter set is done from an inline **"Enable Filters" / "Disable Filters"** button in the bar — **not** from a toggle inside the sheet.

#### Filter Sheet Layout

The sheet is titled **"Filters"** and contains:
- Expandable **cards** — one for the built-in **Timestamp** (labeled "Time"), one for each field in schema order
- Each card header shows the field name and, when collapsed, a summary of any active filter condition (e.g., "= Trail" or "between Mar 1 – Mar 31")
- Expanding a card reveals its operator/value inputs and a **"Clear Filter"** button to remove just that field's condition
- A **"Clear All"** button removes all filter conditions at once
- Multiple field conditions combine with **AND** logic (an entry must satisfy every active condition)

#### Filter Operators by Field Type

**Number and Rating:**
- equals (=)
- not equals (≠)
- greater than or equal (≥)
- less than or equal (≤)
- between (min and max, inclusive)
- is set (has any value)
- is not set (empty/unfilled)

(There are no strict greater-than / less-than operators for Number or Rating — use ≥ / ≤ or "between." Number/Rating equality uses a small tolerance, so decimal values that are essentially equal will match.)

**Duration** and **Distance/Pace** (same numeric operator set):
- equals, not equals, ≥, ≤, between, is set, is not set
- Note: the filter value is interpreted in base units — **seconds** for Duration and **meters** for Distance/Pace — which may differ from the unit shown in the views.

**Text:**
- equals
- not equals
- contains
- starts with
- ends with
- is empty
- is not empty

(Note: **equals / not equals** are case-**sensitive**, while **contains / starts with / ends with** are case-**insensitive**.)

**Toggle:**
- is on (Yes/true)
- is off (No/false)

(These are the only two selectable options — a Toggle is always set, so there is no "is set" / "is not set" for it.)

**Single Select:**
- is [option]
- is not [option]
- is set (has a selection)
- is not set (no selection made)

**Multi Select:**
- includes any of [options]
- includes all of [options]
- excludes [option]
- is set (at least one option checked)
- is not set (no options checked)

**Date/Time:**
- before [date]
- after [date]
- between [date] and [date]
- is set (a date has been entered)
- is not set (no date entered)

The Date/Time (and Timestamp) filter also offers a **quick-interval** builder: a reference picker (**This / Last / Next**) combined with a period (**Day / Week / Month / Year**) — e.g. "This Week" or "Last Month" — with a **Custom…** option to switch to explicit before/after/between date pickers.

**Location:**
- is set (coordinates were captured)
- is not set (no location captured)

**Timestamp** (built-in, always available):
- before [date]
- after [date]
- between [date] and [date]
- is set / is not set (every entry always has a timestamp, so these match everything / nothing)

## Exporting and Backup

LogHound has two distinct data output features — they serve different purposes and are accessed from different places:

| Feature | Purpose | Format | Where to find it |
|---------|---------|--------|-----------------|
| **CSV Export** | Share or analyze data in spreadsheets | .csv | Log list or log detail → "..." menu |
| **.loghound Backup** | Full backup and restore of logs between devices or as a safety copy | .loghound (JSON) | Settings only |

**Key distinction**: CSV Export is for getting your data *out* into other tools. Backup/Restore is for moving or protecting your *entire log* (schema + all entries). CSV files cannot be re-imported into LogHound — only .loghound backup files can.

### CSV Export

Available from:
- Log list → "..." button on a log → **CSV Export**
- Log detail view → "..." menu → **CSV Export**
- Both open the same flow

**Output format**:
- First row: headers — "Timestamp" followed by each field name in schema order
- Subsequent rows: one per entry, sorted by timestamp (newest first)
- Timestamp format: always **ISO 8601 in UTC** (e.g., `2026-03-15T13:45:00Z`). Note this is UTC, not the local time shown in the app — exported times can look shifted by your timezone offset. Date/Time field cells use the same UTC ISO format.
- **Export ignores any active filter** — the CSV always contains **all** of the log's entries, even if the views are currently showing a filtered subset.
- **Text**: value as-is
- **Number**: the numeric value; if the field has "Allow decimals" turned **off**, it is exported as a whole integer (truncated), otherwise with its stored decimals
- **Toggle**: "Yes" or "No" by default (these display strings are configurable in the exporter)
- **Single Select**: the option text, or empty if not set
- **Multi Select**: selected option texts joined with " | " by default, or empty
- **Date/Time**: formatted date/time or empty
- **Duration**: a numeric value in the field's configured Table & Summary unit (e.g., minutes) — **not** the H:MM:SS display form — or empty
- **Rating**: numeric value (e.g., 3), or empty
- **Location**: exported as **separate columns** — `<name> Lat` and `<name> Lon`, plus optional `<name> Alt`, `<name> ±Horiz`, and `<name> ±Vert` columns when those options are enabled (not a single combined cell)
- **Distance/Pace**: exported as **three columns** — distance (header shows the configured distance unit), **time always in seconds** (header `<name> Time (sec)`, regardless of the field's display unit), and pace/speed (header reflects the configured units). Distance and pace/speed are written to 6 decimal places, time to 3.

**Share Sheet**: After generating, the iOS Share Sheet opens. Available destinations: AirDrop, email, Files app, iCloud Drive, third-party apps that accept files.

**Disabled when**: The log has no entries (no data to export).

### .loghound Backup

**What it is**: A `.loghound` file that packages one or more complete logs — schema, all entries, all configurations — into a single portable file. Under the hood it is a **JSON** document. Entry values inside it are keyed by **field name**, so on restore values are matched back to fields **by name** — a caveat worth knowing if a schema has two fields with the same name.

**How to create**:
1. Settings → tap **Backup Logs**
2. A list of all active logs appears with checkboxes
3. Select desired logs (individual checkboxes or **Select All**)
4. Tap **Backup**
5. Share Sheet opens

**What's included in the file**:
- Complete field schema for each selected log
- Every entry with all field values
- All field configurations (options lists, deprecated options, etc.)
- Log metadata (name, sort order)

**Stashed and trashed logs**: Are not available in the backup selection — only active logs are shown.

### Restoring from a .loghound File

**How to restore**:
1. Settings → tap **Restore Logs**, OR
2. Tap a .loghound file from Files app / email attachment → LogHound opens and handles the import

**Confirmation**: A sheet shows "Restore X log(s) from this file?" with the list of log names. The user taps Restore to proceed.

**Name conflict resolution**: If a log in the backup has the same name as an existing **active** log, the imported log is automatically renamed by appending a space and a number starting at 2: "My Log" → "My Log 2". If "My Log 2" also exists, it becomes "My Log 3", and so on. (Conflicts are only checked against active logs, not stashed or trashed ones.) A success message lists all renames.

**Result**: Restored logs are **added** to the active log list alongside existing logs. No existing logs are overwritten or deleted.

**Stashed/trashed logs**: If a user's active log list includes stashed logs they want to restore from backup — they should restore from backup and the imported log will be active. The stashed version remains in the Stash separately.

## Common Questions and Answers

**Q: Can I change a field's type after creating it?**
A: No. Field types are permanent. The user must delete the field (losing any existing data in that field) and add a new one with the desired type.

**Q: I deleted a field by accident — can I get the data back?**
A: No. Field deletion is permanent and immediate. The data cannot be recovered. Encourage the user to use Backup regularly to protect against this.

**Q: Can I use LogHound on iPad?**
A: LogHound is an iOS app and runs on iPad via iOS compatibility. The interface is designed for iPhone but functions on iPad.

**Q: Does LogHound sync between devices?**
A: No. There is no cloud sync. Data lives only on the device where it was entered. To move data between devices, use the .loghound backup and restore workflow.

**Q: Why doesn't my field appear in Summary view?**
A: Summary view only shows Number, Rating, Toggle, Duration, and Distance/Pace fields. Text, Single Select, Multi Select, Date/Time, and Location fields are excluded. If the log has none of those aggregatable types, Summary view shows an empty state.

**Q: Why is the "..." log action menu showing fewer options than expected?**
A: CSV Export is disabled when the log has no entries. All other actions are always available.

**Q: I see old values pre-filled in a new entry form — is this a bug?**
A: No. This is the **draft** feature. The user previously started filling an entry, dismissed the form without saving, and the values were saved as a draft. To clear the draft, they can tap Cancel → Discard (if prompted) or just save the form.

**Q: Can I import data from other apps?**
A: Not directly. LogHound does not import CSV files — it only exports them. Data must be entered manually or restored from a .loghound backup created by LogHound itself.

**Q: Why are some options not showing in my Single Select field?**
A: Options may have been **deprecated**. Deprecated options are hidden from the picker but preserved in existing entries. To see all options including deprecated ones, go to the schema editor → tap the field → view the options list.

**Q: I bought LogHound on my old phone but it shows "Trial Expired" on my new phone.**
A: Tap **Restore Purchases** on the paywall or in Settings. This re-validates the purchase with Apple's servers and unlocks the app.

**Q: Can I track something that happens multiple times a day?**
A: Yes. Just create multiple entries in the same log on the same day. Each entry has its own timestamp. Filters and Summary view can aggregate or filter by date range.

**Q: What happens to my filters when I close the app?**
A: Filters **persist** between sessions. If you set a filter and close the app, the filter is still active when you return. To clear filters, open the Filter sheet and tap "Clear All" or toggle the master switch off.

**Q: Can I have different fields in different logs?**
A: Yes. Each log has its own completely independent schema. A Running log can have Distance, Duration, and Route while a Medication log has Dose, Time, and Notes. There is no shared schema system.

**Q: What is the entry limit per log?**
A: There is no documented per-log entry limit. Performance depends on device storage and available memory.

**Q: How accurate is the GPS distance/pace tracking? My distance looks off.**
A: Distance/Pace relies **entirely on GPS**, and GPS is approximate. The app measures flat, horizontal distance between GPS points, so it doesn't account for elevation on **hilly** routes, and in **urban** areas (and from normal GPS jitter) signal reflections can add spurious wiggle that **overestimates** distance; weak signal can occasionally undercount it. Treat the numbers as good estimates, not exact measurements — the error is usually largest in hilly and built-up areas. For an exact figure, use a plain **Number** field and enter it manually. Also note elapsed time keeps running while you're stopped (no auto-pause), which affects pace.

**Q: Why can't I name a field "Time" or "Location"?**
A: Those names (and `timestamp`, `latitude`, `longitude`, `lat`, `lng`, `lon`, `coordinates`, `created_at`) are reserved for the app's built-in timestamp and location internals. Pick a slightly different name like "Start Time" or "Where".

**Q: My exported CSV times don't match what I see in the app.**
A: CSV timestamps are written in **UTC** (ISO 8601), while the app displays your **local** time — so exported times can look shifted by your timezone offset. The data is correct; it's just in UTC.

**Q: Does the CSV respect my current filter?**
A: No. CSV export always includes **all** entries in the log, regardless of any filter active in the views.

## Use Case Schema Recipes

These are recommended field configurations for common tracking needs. Share these with users who ask "how would I track X in LogHound?"

### Medication / Supplement Log
- **Medication** — Single Select (options: each medication name)
- **Dose** — Number (allow decimals: off, for whole pill counts) or Text (for "500mg")
- **Time Taken** — Date/Time
- **Side Effects** — Multi Select (options: nausea, headache, drowsiness, none, etc.) or Text
- **Notes** — Text

### Fitness / Workout Log
- **Exercise Type** — Single Select (options: Running, Cycling, Weights, Yoga, etc.)
- **Duration** — Duration (hours/minutes/seconds, with an optional built-in timer)
- **Distance** — Distance/Pace (GPS-tracked distance, time, and pace) or Number if entering manually
- **Intensity** — Rating (emoji: 🔥, max: 5)
- **Heart Rate** — Number (bpm, allow decimals: off)
- **Notes** — Text

### Habit Tracker
- **Habit** — Single Select (if tracking multiple habits in one log) OR use a separate log per habit
- **Completed** — Toggle (this is the key field; Summary → Sum + Week shows total completions)
- **Duration** — Duration (if tracking time-based habits)
- **Quality** — Rating (if tracking quality of completion)
- **Notes** — Text

### Sleep Log
- **Bedtime** — Date/Time
- **Wake time** — Date/Time
- **Hours slept** — Number (allow decimals: on, e.g., 7.5)
- **Quality** — Rating (emoji: 😴 or ⭐)
- **Dreams** — Toggle
- **Notes** — Text

### Mood / Mental Health Diary
- **Mood** — Rating (emoji: 😊 or ❤️, max: 5)
- **Energy** — Rating (emoji: ⚡, max: 5)
- **Anxiety** — Rating (emoji: 😰, max: 5)
- **Activities** — Multi Select (options: exercise, socializing, work, nature, etc.)
- **Sleep hours** — Number (allow decimals: on)
- **Notes** — Text

### Food / Nutrition Log
- **Meal** — Single Select (Breakfast, Lunch, Dinner, Snack)
- **Food** — Text
- **Calories** — Number (allow decimals: off)
- **Protein** — Number (grams, allow decimals: off)
- **Satisfaction** — Rating (emoji: 😋)

### Reading Log
- **Title** — Text (required)
- **Author** — Text
- **Genre** — Single Select (Fiction, Non-fiction, Biography, etc.)
- **Rating** — Rating (emoji: ⭐, max: 5)
- **Pages read** — Number (allow decimals: off)
- **Started** — Date/Time
- **Finished** — Date/Time
- **Notes** — Text

### Car Maintenance Log

## Data Loss Prevention Guide

Help users avoid losing data. These are the scenarios that can cause permanent data loss:

1. **Deleting a field**: All data in that field across all entries is permanently deleted. No undo. Recommendation: only delete fields you are sure you don't need.

2. **Deleting an entry**: Permanent, no trash. Recommendation: use Duplicate if you want to modify an entry while preserving the original.

3. **Emptying trash**: Logs in trash are permanently deleted. Recommendation: restore important logs from trash before using Empty Trash.

4. **Reset All Data**: Nuclear option, no recovery. Should only be used to completely start over.

5. **No backup**: If the device is lost, damaged, or reset without a .loghound backup, all data is gone. Recommendation: regular backups via Settings → Backup Logs, stored to iCloud or emailed to yourself.

6. **Select options are safe to remove**: "Removing" a Single/Multi Select option only deprecates it (moves it to "Removed Choices"); it stays visible in existing entries and can be restored. There is no hard-delete that erases historical values, so removing an option does not cause data loss.

## Troubleshooting Reference

| Problem | Likely Cause | Solution |
|---------|-------------|----------|
| Data seems missing | Log was stashed or deleted | Check Stash (bottom-left toolbar) and Trash (bottom-right toolbar) |
| Form pre-filled with old data | Draft was saved from previous session | Fill correctly and Save, or tap Cancel → Discard Draft |
| Save button greyed out | Required field not filled | Look for field with red asterisk (*) |
| Field not in Summary view | Wrong field type | Only Number, Rating, Toggle, Duration, Distance/Pace appear in Summary |
| Option not in Select picker | Option was deprecated | Check schema editor → field → options list for deprecated options |
| Purchased but trial expired message | Purchase not validated on this device | Tap Restore Purchases (paywall or Settings) |
| Log not in active list | Log was stashed | Check Stash view |
| Restore overwrote my log | It didn't — restore adds logs, never overwrites | The original and restored logs coexist (restored may be renamed) |
| Summary showing no data | No aggregatable fields | Add a Number, Rating, Toggle, Duration, or Distance/Pace field to the schema |
| CSV export greyed out | Log has no entries | Add at least one entry first |
